## 2. Database Design

We will use a **Polyglot Persistence** approach:
*   **MySQL:** For structured financial data (ACID compliance is crucial for money)
*   **MongoDB:** For flexible, high-volume audit logs

### 2.1 MySQL Schema

#### Table 1: `users` (Authentication)
```sql
CREATE TABLE users (
    user_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    role ENUM('ADMIN', 'AUDITOR', 'OPERATOR') DEFAULT 'OPERATOR',
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### Table 2: `transactions` (System Records / Master Data)
```sql
CREATE TABLE transactions (
    transaction_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    reference_number VARCHAR(50) UNIQUE NOT NULL,  -- Unique Key for matching
    amount DECIMAL(15, 2) NOT NULL,
    transaction_date DATE NOT NULL,
    description VARCHAR(255),
    status ENUM('PENDING', 'MATCHED', 'MISMATCHED', 'DUPLICATE') DEFAULT 'PENDING',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by BIGINT,
    FOREIGN KEY (created_by) REFERENCES users(user_id)
);
```

#### Table 3: `upload_batches` (Grouping uploaded files)
```sql
CREATE TABLE upload_batches (
    batch_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    filename VARCHAR(255) NOT NULL,
    upload_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    total_records INT,
    matched_count INT DEFAULT 0,
    mismatched_count INT DEFAULT 0,
    uploaded_by BIGINT,
    FOREIGN KEY (uploaded_by) REFERENCES users(user_id)
);
```

#### Table 4: `reconciliation_results` (Linking System vs Upload)
```sql
CREATE TABLE reconciliation_results (
    result_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    batch_id BIGINT,
    transaction_id BIGINT,  -- Reference to system record
    uploaded_reference VARCHAR(50),
    uploaded_amount DECIMAL(15, 2),
    status ENUM('MATCHED', 'AMOUNT_MISMATCH', 'DATE_MISMATCH', 'NOT_FOUND', 'DUPLICATE'),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (batch_id) REFERENCES upload_batches(batch_id),
    FOREIGN KEY (transaction_id) REFERENCES transactions(transaction_id)
);
```

### 4.2 Key Service Logic

#### A. File Upload Service (Pseudo-code)
```java
@Service
public class FileUploadService {

    public Batch processFile(MultipartFile file, User user) {
        // Step 1: Validate file type (CSV or XLSX)
        if (!isValidFile(file)) {
            throw new InvalidFileException("Only CSV/Excel allowed");
        }

        // Step 2: Parse file (Using Apache POI or OpenCSV)
        List<TransactionDTO> uploadedData = parser.parse(file);

        // Step 3: Save batch record to MySQL
        Batch batch = saveBatchMetadata(file.getOriginalFilename(), user);

        // Step 4: Trigger async processing
        reconciliationEngine.processAsync(batch.getId(), uploadedData);

        return batch;
    }
}
```

#### B. Matching Engine (The Core Logic)
```java
@Service
public class MatchingEngine {

    public void processReconciliation(Long batchId, List<TransactionDTO> uploadedData) {
        for (TransactionDTO uploadedTx : uploadedData) {
            // 1. Find exact match in MySQL by reference number
            Transaction systemTx = transactionRepo.findByRefNumber(uploadedTx.getRef());

            if (systemTx == null) {
                // NOT FOUND in system
                saveResult(batchId, uploadedTx, "NOT_FOUND");
            } else if (systemTx.getAmount().equals(uploadedTx.getAmount())) {
                // PERFECT MATCH
                saveResult(batchId, systemTx, uploadedTx, "MATCHED");
            } else {
                // AMOUNT MISMATCH
                saveResult(batchId, systemTx, uploadedTx, "AMOUNT_MISMATCH");
            }

            // 2. Log EVERYTHING to MongoDB
            auditService.log("RECONCILIATION", user, "BATCH", batchId);
        }
    }
}
```

#### C. Audit Service (MongoDB Logging)
```java
@Service
public class AuditService {

    @Autowired
    private AuditLogRepository auditRepo;

    public void log(String action, User user, String entityType, String entityId) {
        AuditLog log = new AuditLog();
        log.setAction(action);
        log.setUserId(user.getUserId());
        log.setUserName(user.getUsername());
        log.setEntityType(entityType);
        log.setEntityId(entityId);
        log.setTimestamp(LocalDateTime.now());
        log.setIpAddress(getClientIP()); // From HttpServletRequest

        auditRepo.save(log); // Saves to MongoDB
    }
}
```

### 5.2 Spring Security Configuration
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public JwtAuthenticationFilter jwtFilter() {
        return new JwtAuthenticationFilter(jwtUtil);
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtFilter(), UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

## 9. Key Design Decisions & Justification

| Decision | Rationale |
| :--- | :--- |
| **MySQL for Transactions** | Financial data requires ACID compliance (Atomicity). We cannot have half-matched transactions. |
| **MongoDB for Audit** | Audit logs are unstructured, can have variable fields, and need high write throughput. |
| **JWT for Auth** | Stateless authentication scales better for REST APIs than session-based auth. |
| **Async Processing** | File reconciliation can take time. Processing it asynchronously (using `@Async` or Queues) prevents API timeouts. |
| **Reference Number as Key** | Using a unique business key (Reference #) rather than internal IDs for matching. |
